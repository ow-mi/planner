# Feature Specification: Solver UI Integration

**Feature Branch**: `001-solver-ui-integration`  
**Created**: 2024-12-19  
**Status**: Draft  
**Input**: User description: "Add a new spec for using the solver python tool with the alpine.js ui"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Run Solver with UI Data and Configuration (Priority: P1)

A user wants to execute the planning solver directly from the web UI using the CSV data they've uploaded and edited, along with the priority configuration settings they've configured in the Configuration tab.

**Why this priority**: This is the core functionality that connects the UI to the solver. Without this, users cannot execute planning runs from the UI, which is the primary value proposition of the integration.

**Independent Test**: Can be tested independently by verifying that clicking a "Run Solver" button in the UI triggers the solver execution with the correct data and configuration, and displays results or errors appropriately.

**Acceptance Scenarios**:

1. **Given** a user has uploaded CSV files in the Input Data tab, **When** they click "Run Solver" in the Solver tab, **Then** the system extracts the CSV data from the edited tables, packages it with the configuration JSON from the Configuration tab, executes the solver, and displays the execution status
2. **Given** a user has configured priority settings in the Configuration tab, **When** they run the solver, **Then** the system uses the JSON output from the Configuration tab as the priority_config.json file for the solver
3. **Given** the solver execution completes successfully, **When** results are available, **Then** the system displays the solution status, makespan, and provides access to output CSV files in the Output Data tab
4. **Given** the solver execution fails or encounters errors, **When** an error occurs, **Then** the system displays clear error messages indicating what went wrong (missing files, validation errors, solver errors, etc.)

---

### User Story 2 - Configure Solver Parameters (Priority: P2)

A user wants to set solver execution parameters (time limit, debug level, output folder) before running the solver from the UI.

**Why this priority**: Users need control over solver execution parameters to optimize performance and debugging. This enables users to customize solver behavior without modifying command-line arguments.

**Independent Test**: Can be tested independently by verifying that solver parameter inputs in the UI are correctly passed to the solver execution and affect the solver's behavior accordingly.

**Acceptance Scenarios**:

1. **Given** a user is in the Solver tab, **When** they set a time limit value, **Then** the system uses that time limit when executing the solver
2. **Given** a user selects a debug level from a dropdown, **When** they run the solver, **Then** the system uses that debug level for logging
3. **Given** a user specifies an output folder path, **When** they run the solver, **Then** the system saves results to that specified folder instead of the default location

---

### User Story 3 - View Solver Execution Progress and Results (Priority: P2)

A user wants to see real-time progress updates during solver execution and view the results once the solver completes.

**Why this priority**: Users need feedback during long-running solver executions to understand that the system is working and to monitor progress. Results viewing is essential for users to access the planning output.

**Independent Test**: Can be tested independently by verifying that progress indicators update during execution and that results are properly displayed when available.

**Acceptance Scenarios**:

1. **Given** a user has started a solver execution, **When** the solver is running, **Then** the system displays progress indicators (e.g., "Running...", elapsed time, current phase)
2. **Given** the solver completes successfully, **When** results are available, **Then** the system displays key metrics (solution status, makespan, number of scheduled tests) and provides links or previews of output CSV files
3. **Given** the solver is still running, **When** a user navigates away from the Solver tab, **Then** the execution continues in the background and results are available when they return

---

### User Story 4 - Export and Download Results (Priority: P3)

A user wants to download the solver output files (CSV reports, logs) generated by the solver execution.

**Why this priority**: Users need to save and share solver results. This enables users to archive results, share with team members, or use results in other tools.

**Independent Test**: Can be tested independently by verifying that download buttons or links correctly retrieve and download the generated output files from the solver execution.

**Acceptance Scenarios**:

1. **Given** a solver execution has completed successfully, **When** a user clicks "Download Results" in the Output Data tab, **Then** the system packages all output CSV files and logs into a downloadable archive or provides individual file downloads
2. **Given** output files are available, **When** a user requests a specific file download, **Then** the system provides that file for download in its original format

---

### Edge Cases

- What happens when required CSV files are missing from the uploaded data?
- How does the system handle invalid or malformed CSV data when preparing to run the solver?
- What happens when the JSON configuration from the Configuration tab is invalid or incomplete?
- How does the system handle solver timeouts or very long-running executions?
- What happens when the solver returns an infeasible solution?
- How does the system handle concurrent solver executions if a user starts multiple runs?
- What happens when disk space is insufficient for saving output files?
- How does the system handle network or connectivity issues if the solver runs on a remote server?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST allow users to execute the planning solver from the Solver tab in the UI
- **FR-002**: System MUST extract CSV data from the edited tables in the Edit Data tab when preparing solver input
- **FR-003**: System MUST use the JSON configuration output from the Configuration tab as the priority_config.json file for solver execution
- **FR-004**: System MUST package all required CSV files (data_legs.csv, data_test.csv, data_fte.csv, data_equipment.csv, data_test_duts.csv) from the UI data before executing the solver
- **FR-005**: System MUST validate that all required CSV files are present before allowing solver execution
- **FR-006**: System MUST allow users to configure solver parameters (time limit, debug level, output folder) in the Solver tab
  - **Note**: The output_folder parameter specifies the solver's temporary working directory for execution. Results are returned to the UI for local export and are not persisted on the server per FR-018.
- **FR-007**: System MUST display solver execution status and progress indicators during execution
- **FR-008**: System MUST display solver results (solution status, makespan, key metrics) in the Output Data tab after successful execution
- **FR-009**: System MUST display clear error messages with specific error categories and actionable guidance when solver execution fails, including validation errors, missing files, or solver errors
- **FR-010**: System MUST provide access to generated output CSV files and logs in the Output Data tab
- **FR-011**: System MUST allow users to download solver output files individually or as a package
- **FR-012**: System MUST handle solver execution asynchronously so the UI remains responsive during long-running executions
- **FR-013**: System MUST preserve uploaded and edited CSV data in the UI during solver execution by maintaining data in Alpine.js component state (x-data) throughout the execution lifecycle
- **FR-014**: System MUST validate CSV data format and required columns before executing the solver
- **FR-015**: System MUST use a backend API service architecture for solver execution
- **FR-016**: System MUST handle solver timeouts gracefully by returning partial results when available
- **FR-017**: System MUST queue concurrent solver executions with single active execution to prevent resource contention
- **FR-018**: System MUST return solver results to the UI for local export rather than persisting on the server

### Key Entities

- **Solver Execution**: Represents a single run of the planning solver with specific input data, configuration, and parameters. Contains execution status, start time, end time, results, and output file references.
- **Input Data Package**: Collection of CSV files (data_legs.csv, data_test.csv, data_fte.csv, data_equipment.csv, data_test_duts.csv) extracted from the UI's edited tables, prepared for solver execution.
- **Configuration Package**: JSON configuration file (priority_config.json) derived from the Configuration tab output, containing priority rules, deadlines, and optimization settings.
- **Solver Results**: Output from solver execution including solution status, makespan, test schedules, resource utilization data, and generated CSV report files.
- **Execution Parameters**: User-configurable settings for solver execution including time limit (seconds), debug level (DEBUG/INFO/WARNING/ERROR/CRITICAL), and output folder path.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Users can successfully execute the solver from the UI with uploaded CSV data and configuration settings in under 30 seconds of setup time
- **SC-002**: Solver execution completes and displays results for 95% of valid input configurations without errors
- **SC-003**: Users receive clear error messages within 5 seconds when solver execution fails due to invalid data or configuration
- **SC-004**: Solver execution progress is visible to users within 2 seconds of starting execution
- **SC-005**: Users can access and download solver output files within 10 seconds of solver completion
- **SC-006**: The UI remains responsive and usable during solver execution, with no blocking operations that prevent user interaction with other tabs
- **SC-007**: All required CSV files are correctly extracted and packaged from the UI data before solver execution in 100% of cases

## Assumptions

- The solver Python tool (`planner_v4.main`) is available and executable from the system running the UI
- CSV files uploaded and edited in the UI follow the expected format and column structure required by the solver
- The JSON configuration output from the Configuration tab matches the expected priority_config.json schema
- Solver execution may take several minutes for complex planning scenarios, requiring asynchronous execution
- Output files generated by the solver follow predictable naming conventions and formats
- Users have sufficient disk space and permissions to save solver output files
- The UI communicates with solver through a backend API service
- Solver results are returned to the UI for local export rather than persisted on the server
- The output_folder parameter in solver configuration refers to the solver's temporary working directory during execution, not a persistent storage location. All results are returned to the UI for local export per FR-018.

## Dependencies

- Existing Alpine.js UI (`config_editor.html`) with Input Data, Edit Data, and Configuration tabs
- Python solver tool (`planner_v4.main`) with command-line interface support
- CSV parsing and editing functionality already implemented in the UI
- JSON configuration generation already implemented in the Configuration tab

## Out of Scope

- Modifying the solver Python tool itself (only integration is in scope)
- Real-time collaborative editing of CSV data during solver execution
- Version control or history tracking of solver executions
- Scheduling or queuing multiple solver executions
- Integration with external data sources or APIs beyond the UI
- Modifying the solver's optimization algorithms or constraint logic
